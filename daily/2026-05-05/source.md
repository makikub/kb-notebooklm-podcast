# 今日のランダムテーマ

## Orchestration Patterns FAQ：マルチエージェント調整を「実務の制御フロー」として捉え直す

今回のテーマは、マルチエージェントの「オーケストレーション・パターン」です。

一見すると、これは「複数のAIエージェントをどう組み合わせるか」という抽象的な分類の話に見える。しかし実務で重要なのは、エージェントの人数ではなく、**誰が次の動きを決めているのか**、**どの単位で仕事が分解されるのか**、**どこで検証が入るのか**、そして**イベントに反応する仕組みなのか、親が指示する仕組みなのか**という制御フローの違いである。

KBの中心ノートである「Orchestration Patterns FAQ」は、まさにこの混乱をほどくための実務的なFAQになっている。特に重要なのは、以下の見取り図である。

- オーケストレーター型：親エージェントやワークフローエンジンが分解・委任・統合する
- ジェネレーター・ベリファイア型：実装する主体と検証する主体を分ける
- メッセージバス型：イベントが発行され、関心のあるエージェントやワーカーが反応する
- TAKTのような宣言的ワークフロー：YAMLなどでステップと遷移を定義し、中央管理で進める

このテーマは、2026年のエージェント型コーディングの文脈とも強くつながっている。Anthropicのトレンドレポートでは、開発者の役割が「コードを直接書く人」から「エージェントを調整し、出力を評価し、アーキテクチャやステークホルダー適合性を守る人」へ移ると整理されている。つまり、今後の実務では「どのモデルを使うか」だけでなく、**どの調整パターンで仕事を流すか**が重要な設計論点になる。

# なぜPodcast向きか

このテーマがポッドキャスト向きなのは、単なる技術分類ではなく、現場で起きる誤解や設計判断が多いからである。

たとえば「マルチエージェント」と聞くと、多くの人はすぐに「プランナー、実装者、レビュアー、セキュリティ担当、ドキュメント担当……」のようなフルチーム構成を想像しがちである。しかしFAQは、そこに冷静な補助線を引く。

実際には、すべての役割を別エージェントに分ける必要はない。むしろ実務では、**検証だけを分離する**ほうが現実的なことが多い。

- メインエージェントが計画と実装を行う
- ベリファイアエージェントが明示的な基準に照らしてレビューする
- 親エージェントまたはメインワーカーが、受け入れ・修正・再試行を判断する

これは、フル装備の多職種チームというより、**ジェネレーター・ベリファイア・ループ**に近い。会話で扱うと、「マルチエージェントって本当に必要なの？」という素朴な疑問から、「どこだけ分けると効果が大きいのか？」という実務的な議論へ自然に展開できる。

また、オーケストレーター型とメッセージバス型の違いも、比喩で説明しやすい。

- オーケストレーター：マネージャーが担当者に仕事を割り振る
- メッセージバス：共有インシデントボードにイベントが貼られ、専門家が自分に関係するものを拾う

この対比は、音声で聞いても理解しやすい。さらに、TAKTの位置づけや、プロダクト開発における主ワークフローと支援自動化の分担まで話を広げられる。

## 2026年のエージェント型コーディングの流れ

Anthropicの「2026 Agentic Coding Trends Report」は、エージェント型コーディングが、開発者を直接の実装者からエージェントのオーケストレーターへ移行させると予測している。

ただし、その見方は「人間が不要になる」というものではない。レポートの重要なニュアンスは、協働的なフレーミングである。開発者は仕事の約60%でAIを使う一方、完全に委任できるタスクは0〜20%にとどまる、とされている。勝ち筋は、人間をループから外すことではなく、**人間の専門性を重要な箇所に集中させること**だとされる。

この文脈では、複数エージェントの調整は単なる先端的な実験ではない。実装速度が上がるほど、計画、検証、レビュー、承認、統合の設計が重要になる。つまり、エージェントを増やすこと自体ではなく、**レビュー能力をどう保つか**が中心課題になる。

# 中心アイデア

## 1. オーケストレーター型とは何か

オーケストレーター型は、親エージェントまたはワークフローエンジンが、作業を分解し、境界づけられたサブタスクを割り当て、結果を集め、次に何が起こるべきかを決めるパターンである。

これはプロダクト開発において直感的である。なぜなら、人間のプロジェクト管理に似ているからだ。

- ひとりの所有者が全体を把握する
- 仕事を小さく分ける
- 担当者に委任する
- 成果をチェックする
- 統合する
- 必要なら修正や再試行を命じる

この構図は、複雑な開発タスクを扱うときに理解しやすい。特に、作業が比較的きれいに分解できる場合には、まずオーケストレーター・サブエージェント型から始めるのが実務的な出発点になる。

## 2. すべてを別エージェント化しなくてよい

FAQの実務的な価値は、「マルチエージェント＝たくさんの専門家エージェントを作ること」という思い込みを抑制している点にある。

役割分離は有効だが、分けすぎると調整コストが増える。関連ノート「Multi-Agent Orchestration」でも、複数エージェントは専門性を高める一方で、調整複雑性、レイテンシ、トークンオーバーヘッドを増やすと整理されている。

そのため、実務では狭い分離で十分な場合が多い。特に有効なのが、**検証だけを分ける**設計である。

- メインエージェント：計画と実装
- ベリファイアエージェント：明示的基準に照らしたレビュー
- 親またはメインワーカー：受け入れ、修正、再試行の判断

これは「全役割分解」ではなく、生成と検証の分離である。自己評価の弱さを補いながら、チーム構成を過度に複雑化しない。

## 3. メッセージバス型は「人数」ではなく「制御フロー」が違う

メッセージバス型は、親が直接指示するのではなく、イベント駆動で動く。

オーケストレーター型では、親がこう指示する。

- レビュアー、このPRを見てください
- セキュリティエージェント、この変更を確認してください

一方、メッセージバス型では、イベントが発行される。

- `pr_opened`
- `ci_failed`
- `security_alert`

そして、関心のあるエージェントやワーカーがそのイベントを購読し、反応する。

ここで重要なのは、エージェント数ではない。同じように複数エージェントが存在していても、制御フローが違う。

- オーケストレーター = 直接指示の流れ
- メッセージバス = 公開イベントの流れ

この違いをつかまないと、「エージェントが複数いればメッセージバス的」と誤解してしまう。

## 4. メッセージバスにはアクティブなイベント消費層が必要

メッセージバス型では、何らかのイベント消費層がアクティブである必要がある。ただし、それは必ずしも、ひとつのエージェントが `while true` で常に考え続けているという意味ではない。

実装としては、以下のような形がありうる。

- hooks
- webhooks
- queue consumers
- pub/sub subscribers
- cron-based pollers
- lightweight always-on workers

つまり、必要なのは「常に考えているエージェント」ではなく、イベントを受け取り、反応を起動するインフラ層である。この点も、ポッドキャストで誤解を解くのに向いている。

## 5. TAKTは宣言的ワークフロー・オーケストレーターに近い

FAQでは、TAKTは純粋なメッセージバス設計ではなく、宣言的ワークフロー・オーケストレーターとして理解するほうがよいとされている。

理由は以下である。

- YAMLで明示的なステップと遷移を定義する
- plan、implement、review、fixのループを中央ワークフロー制御のもとで実行する
- 複数エージェントや並列レビュアーが存在しても、遷移は中央管理される

したがって、TAKTは以下に近い。

- オーケストレーター・サブエージェント
- ジェネレーター・ベリファイア

一方で、純粋なメッセージバス型とは距離がある。

# 実務での使いどころ

## まずはオーケストレーター・サブエージェント型から始める

作業がきれいに分解できるなら、まずはオーケストレーター型がわかりやすい。親がタスクを分け、結果を統合するため、プロダクト開発の作業感覚に近い。

向いている状況は、たとえば次のようなもの。

- 機能実装を複数のサブタスクに分けられる
- 計画、実装、レビュー、修正の順番がある程度明確
- 中央で進捗と判断を持ったほうが安全
- 成果物の統合責任を明確にしたい

関連ノート「Coding-Agent Harness Patterns」にある「Plan as handoff artifact」とも相性がよい。`plan.md` のようなセッション単位の計画ファイルを、意図の可視化、コンテキストリセット、レビュアー alignment、ベリファイアの批評対象として使える。

## 品質問題が中心ならベリファイアを分ける

「実装速度は出るが、品質や妥当性の確認が不安」という場合、最初に分けるべきは実装者ではなく検証者かもしれない。

ベリファイア分離の利点は次の通り。

- 自己評価バイアスを下げる
- 明示的なレビュー基準を使いやすくする
- フルマルチエージェントより運用が軽い
- 人間レビュアーの前段で問題を減らせる

「Coding-Agent Harness Patterns」でも、完全な多役割分解ではなく、計画レビューや実装レビューだけに第二のエージェントやモデルを使うパターンが挙げられている。これは、評価者の恩恵を安く得る方法である。

## CI、PR、インシデント、リリース周辺にはメッセージバス的反応を加える

プロダクト開発の主ワークフローは、オーケストレーターやエージェントチームで進める。一方で、その周辺の支援自動化には、メッセージバス型が向いている。

FAQでは、実務的な分担として以下が挙げられている。

- 主なプロダクト作業：オーケストレーター・サブエージェントまたはエージェントチーム
- 支援自動化：CI、PR、インシデント、リリースイベント、ドキュメント更新、アラートへのメッセージバス的反応

これはかなり重要な整理である。メッセージバス型を「全部の仕事をイベント駆動にする設計」としてではなく、主ワークフローの周囲に置く支援レイヤーとして使うと考えると、導入しやすい。

## 決定論的チェックとレビュー・ループを近くに置く

「Coding-Agent Harness Patterns」では、lint、hooks、pre-commit checks、testsを実装に近い場所で走らせることが重視されている。これは、エージェントの出力を人間が後で全部見るのではなく、エラー検出を前倒しし、レビュアー負担を減らすためである。

ここでも、オーケストレーションは単に「誰がやるか」ではない。どのタイミングで機械的チェックを入れ、どのタイミングでAIレビューを入れ、どのタイミングで人間が判断するかという設計になる。

# 誤解しやすい点・論点

## 誤解1：マルチエージェントは、役割を細かく分けるほどよい

実際には、分けるほど調整コストが増える。レイテンシやトークンオーバーヘッドも増える。最も有用な分割は、フルロール分解ではなく、検証の分離のような狭い分割であることが多い。

問い：

- その分離は品質を上げるのか、それとも連絡コストを増やすだけか？
- 役割を増やす前に、レビュー基準やテストを明確にできないか？
- ひとつのエージェントで計画と実装を持ち、検証だけ別にするほうがよくないか？

## 誤解2：メッセージバスは、エージェントがたくさんいるシステムのこと

違う。メッセージバスの本質は、イベント発行と購読である。

対立軸はこうである。

| 観点 | オーケストレーター型 | メッセージバス型 |
|---|---|---|
| 制御 | 親が直接指示する | イベントが発行される |
| 比喩 | マネージャーが仕事を割り振る | 共有インシデントボード |
| 反応 | 指名された担当が動く | 関心ある購読者が動く |
| 向き | 分解・統合が必要な主作業 | CI、PR、アラートなどの周辺反応 |

## 誤解3：メッセージバスには、常時思考するAIが必要

必要なのはアクティブなイベント消費層であって、常に思考する単一エージェントではない。hooks、webhooks、queue consumers、pub/sub subscribers、cron-based pollers、lightweight always-on workersなどで実現されうる。

ここは実務上大事である。AIエージェントの設計論と、実行インフラの設計論を混同すると、過剰に複雑な「常駐AI」を作りたくなる。しかしFAQは、その必要はないと整理している。

## 誤解4：TAKTはメッセージバスである

FAQでは、TAKTは宣言的ワークフロー・オーケストレーターに近い。YAMLでステップと遷移を定義し、plan、implement、review、fixのループを中央管理するからである。

複数エージェントや並列レビュアーがいても、遷移を中央で管理しているなら、それはメッセージバスというより、オーケストレーター型やジェネレーター・ベリファイア型に近い。

## 論点：レビュー能力はスケールするのか

Anthropicのトレンドレポートは、開発者がAIを広く使いながらも、完全委任は限定的だとする。つまり、エージェントが増えれば増えるほど、人間の判断やレビューの設計が重要になる。

ここでの問題は、「AIがどれだけ書けるか」ではない。

- どの出力を人間が見るべきか
- どの出力はテストやリンターに任せられるか
- どのレビューはベリファイアエージェントに任せられるか
- どの時点で承認を要求するか
- 組織全体でレビュー能力をどう保つか

これは、マルチエージェントの普及に伴う中心的な運用課題である。

## 基本理解の問い

- オーケストレーター型を一言で言うと何か？
- なぜプロダクト開発ではオーケストレーター型が直感的なのか？
- メッセージバス型は、単にエージェント数が多いだけの設計と何が違うのか？
- TAKTはなぜメッセージバスではなく、宣言的ワークフロー・オーケストレーターに近いのか？

## 実務判断の問い

- 作業がきれいに分解できるなら、どのパターンから始めるべきか？
- 品質チェックが主な課題なら、なぜベリファイア分離が有効なのか？
- いつフルマルチエージェントチームを作るべきで、いつ作るべきではないのか？
- CI失敗、PR作成、セキュリティアラートのようなイベントには、どのパターンが向くのか？

## 対立軸を深める問い

- 「中央制御」と「イベント駆動」は、現場の運用感としてどう違うのか？
- 役割分離による専門化と、調整複雑性の増加はどうトレードオフになるのか？
- 人間のレビュー能力を温存するには、どこを機械化し、どこをAIレビューにし、どこを人間が見るべきか？
- エージェント型コーディングが広がるほど、人間の仕事は楽になるのか、それとも判断責任が濃くなるのか？

## ポッドキャストで使いやすい比喩

- オーケストレーター = プロジェクトマネージャーがタスクを割り振る
- メッセージバス = 共有インシデントボードにイベントが流れてくる
- ベリファイア分離 = 書いた人とは別の人がチェックリストで見る
- ハーネス = モデルを取り巻く作業場、ルール、チェック、引き継ぎ、レビューの仕組み
