# 今日のランダムテーマ：Agent Autonomy（エージェント自律性）

日付：2026-05-03

今回のテーマは **Agent Autonomy**、つまり「エージェントが、人間の即時承認なしに、どの程度まで行動を選び、実行できるか」です。

KB上の定義では、Agent Autonomy は次のように整理されている。

> エージェントが、人間の即時承認なしに、行動を選択し実行できる度合い。

この定義は短いが、実務上は非常に重い。なぜなら、自律性はエージェントの「便利さ」と「リスク」の両方を決めるからである。

自律性が低いエージェントは、安全に見える。人間の確認を何度も挟むからだ。しかし、毎回承認が必要なら、エージェントらしいスピードや連続作業の価値は薄れる。

逆に、自律性が高いエージェントは、複数の作業を連鎖させ、長いワークフローを進められる可能性がある。しかし、失敗したときには、その失敗も高速に進み、人間が途中で止めにくくなる。

このテーマは、単に「AIにどこまで任せるか」という話ではない。より正確には、次の問いに近い。

- どの行動なら自動実行してよいのか
- どの行動では人間の承認が必要なのか
- 信頼が増すにつれて、自律性をどう段階的に上げるべきか
- 判断の一部は、外側のハーネスが担うべきか、それともモデル側に移してよいのか
- ツールアクセスを増やすほど、能力とリスクはどう変わるのか

Agent Autonomy は、Approval Policy、Human in the Loop、Coding Agents、Agent Safety、Harness Engineering、Tool Accessibility と強く結びついている。

# なぜPodcast向きか

このテーマがPodcast向きなのは、単純な賛否では語れないからである。

「エージェントにはもっと自律性を与えるべきだ」という立場もある。  
「いや、承認なしに動かすのは危険だ」という立場もある。

どちらも正しい部分を持っている。

KBでは、Agent Autonomy のトレードオフが次のように表現されている。

- 自律性を高めると、複利的な作業ループを解放できる
- しかし、自律性を高めると、失敗が速くなり、途中で捕捉しにくくなる

この「便利さと危険さが同じレバーで動く」という構造が、会話に向いている。

たとえば、承認プロンプトを増やせば安全になるように見える。しかし Approval Policy のKBでは、次の緊張関係が示されている。

- プロンプトが多すぎると、エージェントは使い物にならなくなる
- プロンプトが少なすぎると、見えないリスクが生まれる
- 静的ルールは監査しやすい
- 一方で、適応的なゲートは高頻度ワークフローで使いやすさを大きく回復できる

つまり、論点は「承認を入れるか入れないか」だけではない。  
「どの承認を、どのタイミングで、どんな粒度で、どの仕組みによって入れるのか」が問題になる。

さらに、Coding Agents の文脈では話が具体化しやすい。コード編集、ターミナルコマンド、テスト、リポジトリ文脈など、エージェントが触る対象がはっきりしているからだ。Coding Agents は、LLMエージェントの中でも特に運用上の要求が厳しく、自律性と監督を考える実験場になりやすい。

# 背景と文脈

## エージェントは「モデル単体」では完結しない

Agent Autonomy を考えるとき、ついモデルの賢さだけに注目してしまう。しかし、KBの Harness Engineering では、エージェントの性能はモデルだけではなく、その周囲の構造にも依存すると整理されている。

Harness Engineering の定義は次の通り。

> エージェントが時間とともにより信頼できる結果を出せるように、環境、成果物、制御、ツール、フィードバックループを設計すること。

つまり、自律性は「モデルに自由を与えるかどうか」だけでは決まらない。

エージェントが何を見られるか。  
どのツールを使えるか。  
どの操作で止められるか。  
どの成果物を人間に渡すか。  
どの時点で検査が入るか。  
どのようなフィードバックを得られるか。

こうした外側の設計が、エージェントの自律性を実質的に決める。

## Human in the Loop は「全部人間が見る」ではない

Human in the Loop は、人間が選ばれた行動やチェックポイントに対して、最終的または定期的な意思決定者として残る運用パターンである。

ここで重要なのは、「常に人間が介入する」という意味ではないことだ。

KBでは、Human in the Loop の価値は、より能力の高いエージェントを、完全な自律行動への信頼なしに実用展開できる点にあるとされている。

一方で、トレードオフも明確だ。

- 人間のチェックポイントは安全性を高める
- しかし摩擦も増やす
- 介入が多すぎると、エージェント的ワークフローの利点を壊す

したがって、Human in the Loop は「人間が毎回止める仕組み」ではなく、「どこに人間の判断を残すべきか」という配置問題として見るべきである。

## Approval Policy は自律性の主要レバー

Approval Policy は、エージェントがどの行動を自動で行ってよく、どの行動に明示的なユーザー承認が必要かを決めるルールシステムである。

これは Agent Autonomy の中核に近い。

ただし、Approval Policy は単なる「許可プロンプトの有無」ではない。KBでは、承認ポリシーには次のようなものも含まれるとされている。

- ブランチ制限
- バックグラウンドジョブの制御
- 停止経路におけるレビューゲート

つまり、自律性の設計は「はい／いいえ」のポップアップだけではなく、作業場所、作業範囲、実行経路、停止条件まで含む。

## Tool Accessibility が自律性の現実的な上限を決める

Tool Accessibility は、人間が使っているツールが、エージェントにとって発見可能で、効果的に使える形で公開されている度合いである。

エージェントに自律性を与えても、必要なツールや文脈にアクセスできなければ、実用性は低いままになる。

一方で、KBでは次の緊張関係も示されている。

- ツールアクセスを増やすと、能力とリスクが同時に増える
- よいインターフェースなしにツールを露出すると、助けになるどころかエージェントを混乱させることがある

これは Agent Autonomy の議論にとって重要である。  
「自律的に動ける」とは、単にモデルが次の手を考えられることではない。必要なツールに触れ、文脈を読み、結果を確認し、必要なら修正できることでもある。

しかし、触れる対象が増えるほど、失敗時の影響範囲も広がる。

# 中心アイデア

## 自律性は単一のスイッチではなく、段階的な設計対象である

Agent Autonomy は「オン／オフ」ではない。

実務では、おそらく以下のような複数の軸で考える必要がある。

- 行動選択の自由度
- 実行できるツールの範囲
- 人間承認が必要な操作の範囲
- 失敗時に止まる仕組み
- 作業結果を人間が確認する頻度
- リポジトリや環境に対する変更権限
- バックグラウンドで進めてよい作業の範囲
- モデルとハーネスの責任分担

KBのOpen Questionsにも、まさに次の問いがある。

- エージェントへの信頼が増すにつれて、自律性をどう段階化すべきか
- モデルの能力が向上するにつれて、どのオーケストレーション判断をハーネスからモデルへ移すべきか

この2つは、Agent Autonomy の最重要論点である。

## 自律性の価値は「複利的な作業ループ」にある

自律性を上げる最大の魅力は、エージェントが単発の回答ではなく、作業ループを回せることにある。

KBでは「compounding work loops」という表現が使われている。日本語では「複利的に積み上がる作業ループ」と表現できる。

これは、エージェントが次のような流れを自分で回せる状態を想像すると理解しやすい。

- 状況を読む
- 次の行動を選ぶ
- ツールを使う
- 結果を確認する
- 必要なら修正する
- 次の作業に進む

ただし、このループが有益に働くには、周囲のハーネス、ツールアクセス、承認ポリシー、人間のチェックポイントが整っている必要がある。

## 自律性のリスクは「失敗の速度」にある

自律性の危険は、単に「失敗するかもしれない」ことではない。  
問題は、失敗が速く、連鎖的に進む可能性があることだ。

人間が毎回承認するなら、失敗はその都度止まりやすい。  
しかし、自律性が高い場合、エージェントは複数の操作を続けて実行できる。便利さと同じメカニズムによって、失敗も進行する。

ここで重要になるのが、Approval Policy と Human in the Loop である。

どの操作は止めるのか。  
どの操作は記録だけでよいのか。  
どの操作は人間に助言的レビューを求めるのか。  
どの操作はブロッキングレビューにするのか。  
どの操作は人間のみ実行にするのか。

Approval Policy のOpen Questionsでは、エスカレーションラダーとして次の選択肢が挙げられている。

- advisory review：助言的レビュー
- blocking review：ブロッキングレビュー
- human-only execution：人間のみ実行

これは、Podcastで深掘りしやすい対立軸である。

# 実務での使いどころ

## 1. Coding Agents の運用設計

Coding Agents は、Agent Autonomy を考えるうえで最も具体的な領域の一つである。

KB上の定義では、Coding Agents は次のように整理されている。

> コード編集、ターミナルコマンド、テスト、リポジトリ文脈を使って、ソフトウェア開発環境で作業するエージェント。

この領域では、エージェントが実行できる行動が多い。

- コードを読む
- コードを編集する
- コマンドを実行する
- テストを走らせる
- リポジトリ文脈を参照する
- 作業結果をまとめる

そのため、自律性の設計が現実問題として現れる。

Coding Agents のトレードオフは明快だ。

- 自律性を上げるとスループットが改善する
- しかし、ミスの影響範囲も広がる
- コード環境は、多くのリスクのある行動を含む
- しかも、エージェントが持つ文脈は常に均一ではない

実務上の問いとしては、次のようなものがある。

- どのコーディングタスクは完全自律でよいのか
- どのタスクは監督付きにすべきか
- どのサブタスクは安価なワーカーモデルに任せるべきか
- どのサブタスクは大きなプランナーモデルやジャッジモデルに任せるべきか
- セッション管理のどの動きはユーザーに明示すべきか
- どの動きはハーネス側で自動化してよいのか

## 2. 承認ポリシーの設計

Agent Autonomy を実務に落とすとき、最初に設計対象になるのが Approval Policy である。

単に「危険そうなものは承認」とすると、プロンプトが多すぎて使いにくくなる。  
逆に「全部自動でよい」とすると、見えないリスクが増える。

ここで有効な見方は、承認ポリシーを「作業の流れを止める場所の設計」として捉えることだ。

検討すべき問い：

- 承認が必要な行動は何か
- 承認不要でよい行動は何か
- 記録だけ残せばよい行動は何か
- ブランチ制限で十分な行動は何か
- バックグラウンドジョブとして許可してよい行動は何か
- 停止経路でレビューすべき行動は何か
- 静的ルールで足りるか
- 適応的な分類器やゲートが必要か

KBでは、静的ルールは監査しやすい一方で、適応的ゲートは高頻度ワークフローの使いやすさを回復できると整理されている。これは、実務上の大きな設計判断である。

## 3. Human in the Loop の配置

Human in the Loop は、自律性を否定するものではない。むしろ、自律性を現実に使うための緩衝材である。

重要なのは、人間をどこに置くかだ。

- 最初に計画だけ確認するのか
- 途中の重要操作だけ確認するのか
- 最後の成果物だけ確認するのか
- 失敗しそうなときだけ介入するのか
- 定期的なチェックポイントを設けるのか

KBのOpen Questionにも「長時間実行されるエージェントワークフローのどこに人間のチェックポイントを置くべきか」という問いがある。

これは、Agent Autonomy を単なる権限設定ではなく、ワークフロー設計として考える視点につながる。

## 4. ツールアクセスの優先順位付け

Tool Accessibility の観点では、エージェントが使えるツールを増やすことが重要になる。しかし、ツールを増やせばよいわけではない。

KBでは、ツールアクセスのOpen Questionsとして次のような問いが挙げられている。

- どのツールからエージェントアクセス可能にすべきか
- 汎用ツールのままでよいのはいつか
- いつ型付きの専用ツールに昇格させるべきか
- どのツールはライブ状態、プレビュー、テストフィードバックを直接公開すべきか
- 組織コンテキストは、点の統合ではなく、ファイルシステムや基盤レイヤーとして公開すべきなのはいつか

Agent Autonomy の観点では、これは「エージェントが何を自分で判断できるようになるか」に直結する。

よいツールアクセスは自律性を支える。  
悪いツールアクセスは、自律性を危険にする。

# 誤解しやすい点・論点

## 誤解1：自律性が高いほど常に良い

自律性は便利さを増やすが、同時にリスクも増やす。  
KBでも、より多い自律性はスループットや複利的な作業ループを解放する一方で、失敗を速くし、止めにくくすると整理されている。

したがって、目標は「最大自律性」ではなく「適切な自律性」である。

## 誤解2：承認を増やせば安全になる

承認は安全性を高めるが、増やしすぎるとエージェントの価値を壊す。

Approval Policy の論点は、承認回数を増やすことではなく、承認の配置を設計することである。

- どこで止めるか
- どこは流すか
- どこは記録だけにするか
- どこは人間のみ実行にするか

## 誤解3：Human in the Loop は自律性の反対語である

Human in the Loop は、自律性の反対ではない。  
むしろ、完全な信頼がない状態でも、より能力の高いエージェントを実用化するための運用パターンである。

重要なのは、人間が常に割り込むことではなく、必要なチェックポイントで意思決定者として残ることである。

## 誤解4：ツールを渡せばエージェントは賢くなる

Tool Accessibility は重要だが、ツール露出には設計が必要である。

KBでは、よいインターフェースなしにツールを公開すると、エージェントを助けるどころか混乱させる可能性があるとされている。

つまり、ツールアクセスは単なる数の問題ではない。  
発見しやすさ、使いやすさ、フィードバックの返し方、ライブ状態やプレビューの提示などが関係する。

## 誤解5：自律性はモデルだけの問題である

Agent Autonomy はモデル能力だけで決まらない。

Harness Engineering の観点では、環境、成果物、制御、ツール、フィードバックループが重要である。自律性は、モデルとハーネスの相互作用として設計される。

# NotebookLMが会話で拾うとよい問い

## 基本の問い

- Agent Autonomy とは何か？
- なぜ自律性は、便利さとリスクの両方を決めるのか？
- 自律性が高いエージェントと低いエージェントでは、何が違うのか？
- 「人間の即時承認なしに動く」とは、実務上どこまでを意味するのか？

## トレードオフの問い

- 自律性を上げると、どんな作業ループが可能になるのか？
- 自律性を上げると、なぜ失敗が速く、止めにくくなるのか？
- 承認プロンプトを増やしすぎると、なぜエージェントが使いにくくなるのか？
- 承認プロンプトを減らしすぎると、どんなリスクが見えなくなるのか？

## Approval Policy に関する問い

- Approval Policy は単なる許可プロンプトなのか？
- ブランチ制限やバックグラウンドジョブ制御も、なぜ承認ポリシーの一部なのか？
- 静的ルールと適応的ゲートは、どちらが実務に向いているのか？
- advisory review、blocking review、human-only execution はどう使い分けるべきか？

- Human in the Loop は、エージェントの自律性を下げるだけなのか？
- 人間のチェックポイントは、長時間ワークフローのどこに置くべきか？
- どの作業は最初に人間が見るべきか？
- どの作業は最後のレビューだけでよいのか？

- なぜ Coding Agents は Agent Autonomy を考える実験場になりやすいのか？
- コード編集、ターミナル、テスト、リポジトリ文脈があると、自律性のリスクはどう変わるのか？

- どのコーディングタスクは監督付きにすべきなのか？
- 安価なワーカーモデルと大きなプランナー／ジャッジモデルの分担はどう考えるべきか？

## Harness Engineering と Tool Accessibility の問い

- 自律性はモデル単体で決まるのか、それともハーネスで決まるのか？
- エージェントの周囲にある環境、成果物、制御、ツール、フィードバックループは何を変えるのか？
- ツールアクセスを増やすと、なぜ能力とリスクが同時に増えるのか？
- よいツールインターフェースとは何か？
- どのツールからエージェントに公開すべきか？

## 将来設計の問い

- エージェントへの信頼が増したとき、自律性はどう段階的に上げるべきか？
- モデルが強くなったとき、どの判断をハーネスからモデルに移してよいのか？
- 逆に、どれだけモデルが強くなっても、ハーネス側に残すべき制御は何か？
- 自律性の「ちょうどよい水準」は、タスクごとにどう変わるのか？
