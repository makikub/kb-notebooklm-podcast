# 今日のランダムテーマ

## Harness Engineering Vendor / Practitioner Comparison  
—「モデル性能」ではなく「周囲の設計」を比べるための地図—

今回のテーマは、**ハーネスエンジニアリングを、ベンダーや実務者ごとにどう捉えているか**を比較することです。

ここでいうハーネスエンジニアリングとは、単に「良いプロンプトを書く」ことではありません。  
エージェントがより信頼できる成果を出すために、その周囲にあるもの――リポジトリ構造、ドキュメント、テスト、リンター、レビュー、権限、実行環境、進捗ログ、引き継ぎ成果物、マルチエージェント構成――を設計することです。

KBの定義では、ハーネスエンジニアリングは次のように整理されています。

> エージェントの周囲にある環境、成果物、制御、ツール、フィードバックループを設計し、時間とともにより信頼できる結果を出せるようにすること。

このテーマが面白いのは、同じ「ハーネス」という言葉を使っていても、各社・各実務者が見ている焦点が少しずつ違うからです。

- OpenAIは、リポジトリ設計や機械的制約、エージェントが認識しやすい文脈を重視する。
- Cursorは、エージェント中心の開発ワークスペースと、訓練環境のブートストラップに注目する。
- Cognition / Devinは、クラウドエージェントの実行基盤や組織導入の問題として見る。
- Anthropicは、長時間動作するエージェント、構造化された引き継ぎ、MCP、スキル、クラウド実行、マルチエージェント評価などを強く語る。
- LangChainは、トレースやミドルウェア、自己検証などによる測定可能な性能改善に寄せる。
- Martin Fowlerは、語彙と概念整理を提供する。
- Gotaは、目的から逆算して何をハーネスに入れるべきかを問う。
- r.kagaya / Astermindsは、ルールファイルを超えたAIネイティブな開発プロセス設計として捉える。
- AlgomaticやAI-DLC系の議論は、チームやプロセス側のハーネスを明示する。
- Mitchell Hashimotoは、日々の個人ワークフローの中での採用パターンを示す。
- Ignorance.aiの整理は、組織内の役割変化や「環境構築」と「エージェント管理」の分離に注目する。

つまり、これは単なるベンダー比較ではありません。  
**AIエージェント時代のソフトウェア開発を、どの層から設計し直すか**という比較です。

# なぜPodcast向きか

このテーマはPodcast向きです。理由は、単純な一覧表ではなく、いくつもの対立軸があるからです。

たとえば、次のような問いが自然に出てきます。

- ハーネスは「開発者が裏側で作るもの」なのか、それとも「プロダクトUIとしてユーザーに見せるもの」なのか。
- 大事なのは、プロンプトやルールファイルなのか、それともテスト、実行環境、レビュー、権限管理なのか。
- マルチエージェント化は本当に必要なのか。それとも、計画者・実装者・評価者の役割分離だけで十分なのか。
- 長時間動くエージェントには、どのような引き継ぎや記憶が必要なのか。
- エージェントの生成速度が人間のレビュー速度を超えたとき、何が壊れるのか。
- ハーネスを増やすことは常に良いのか。それとも、目的に合わない制御は剪定すべきなのか。

また、このテーマは「抽象論」と「実務」が行き来しやすいです。

Martin Fowlerのような概念整理から始めて、OpenAIやAnthropicのような具体的な実装思想へ進めることができます。さらに、Gotaやr.kagayaのような実務寄りの論点に移ると、「では自分たちのチームでは何を設計すべきか」という話になります。

会話としては、次の流れが自然です。

1. まず「ハーネスエンジニアリングとは何か」を短く定義する。
2. 次に「なぜモデル単体ではなく周囲の設計が重要なのか」を話す。
3. そのうえで、各ベンダー・実務者の違いを比較する。
4. 最後に、現場で使うならどこから始めるかを考える。

# 背景と文脈

## モデル性能だけでは足りない

KB全体の中心的な主張は、次の一文に集約できます。

> エージェントの性能は、モデルだけで決まるのではなく、モデルの周囲にあるハーネスに大きく依存する。

ここでいう「周囲」とはかなり広いです。

- リポジトリの構造
- エージェントが読めるドキュメント
- アーキテクチャ制約
- テストやリンター
- 構造チェック
- 実行環境の検証
- ブラウザベースの確認
- AIレビューや評価エージェント
- 進捗ログ
- 計画ファイル
- セッション間の引き継ぎ
- 承認ポリシー
- 人間の介入ポイント
- クラウド実行基盤
- 権限、監査、隔離
- マルチエージェントの役割分担

これらは、モデルに「ちゃんとやって」と頼むだけでは生まれません。  
むしろ、モデルが間違えにくく、間違えたときに検出されやすく、長い作業でも文脈を失いにくく、危険な操作では人間や別の評価機構に止められるようにする設計です。

## 生成速度が上がるほど、レビュー側がボトルネックになる

関連ノート「Agent Deployment Gates and Operational Risk」では、エージェントがコードや設定変更、脆弱性発見を人間の確認速度以上に生み出せるようになると、支配的な失敗は「個別のモデルミス」ではなく、**システム全体のレビュー崩壊**へ移ると整理されています。

この観点では、ハーネスは単なる便利ツールではありません。

- マージ前の構造的ゲート
- デプロイ前の検証ゲート
- 人間へのエスカレーションゲート
- デプロイ後のフィードバックゲート

こうした層がなければ、エージェントの生産性はそのまま運用リスクになります。

この文脈で見ると、各社のハーネス観の違いは、「どの失敗モードを最重要視しているか」の違いにも見えてきます。

# 中心アイデア

## 1. OpenAI：リポジトリをエージェントが使える地図にする

OpenAIの焦点は、リポジトリ設計、機械で強制できる不変条件、エージェントが認識しやすい文脈、継続的なクリーンアップ、そしてエージェント間レビューや委譲です。

特徴的なのは、ドキュメントを単なる説明ではなく、**エージェントのための地図**として扱う点です。

- map-like docs
- custom linters
- structured plans
- architecture constraints
- review loops
- interoperable agent tooling
- subagentsによる明示的なモデル階層ルーティング

ここでのハーネスは、かなり「生産工学」的です。  
人間が読むドキュメントではなく、エージェントが迷わず編集し、制約を破らず、他のエージェントに確認してもらえるような構造を作る。

OpenAI的な見方では、良いリポジトリとは、ただ整理されているリポジトリではありません。  
**エージェントが正しく作業しやすいリポジトリ**です。

## 2. Cursor：ハーネスをプロダクトUIと訓練基盤の両方にする

Cursorは、エージェントファーストなコーディングワークスペース設計と、モデル訓練環境のブートストラップに焦点があります。

特徴的な要素は次の通りです。

- 複数エージェントの可視性
- ローカルとクラウドのハンドオフ
- 複数リポジトリ操作
- レビュー面
- ハーネス測定
- 以前のモデルを使って、後続モデルのための実行可能なRL環境を作る autoinstall loop

Cursorの比較上の重要性は、ハーネスを「裏側の実装詳細」に留めない点です。  
ハーネスの仕組みが、そのままユーザーが触るプロダクトUIにもなり、モデル訓練のインフラにもなる。

これは、ハーネスが単なる補助輪ではなく、製品価値そのものになっていく方向性です。

## 3. Cognition / Devin：クラウドエージェントには実行基盤とガバナンスが必要

Cognition / Devinは、クラウド上で動くエージェントのランタイム基盤と、組織導入の問題を強く見ています。

特徴はかなりインフラ寄りです。

- microVM isolation
- hypervisor snapshotting across async gaps
- warm-pool orchestration
- permission inheritance
- audit trails
- engineering-process redesign

この見方では、クラウドエージェントは単なる「遠隔のCLIラッパー」ではありません。  
長時間・非同期で動き、途中で止まり、再開し、権限を引き継ぎ、監査される存在です。

つまり、必要なのはプロンプトやエディタ連携だけではありません。  
隔離、スナップショット、権限、監査、組織プロセスの設計が必要になります。

## 4. Anthropic：長時間エージェントのための総合ハーネス

Anthropicは、KB上では最も広く、長期的なハーネス設計を語っている存在として整理されています。

焦点は非常に多いです。

- long-running agents
- structured handoffs
- initializer agents
- context resets
- multi-agent evaluator patterns
- cloud execution
- scheduled tasks
- reusable skills
- MCP as the production integration layer
- Claude Designのような専用ワークスペース
- managed-agent memory / dreaming / outcomes
- model-speedで出荷するチーム

特徴的なのは、長い時間軸とセッション間の連続性です。

一回のチャットで完結する作業ではなく、複数セッションにまたがり、途中でコンテキストをリセットし、進捗成果物を残し、別のエージェントや人間が引き継げるようにする。

また、planner / generator / evaluator の分離や、rubric-driven outcomes、環境制御、ツール境界、標準化された認証、リモートツール配布など、かなり運用に近い論点も含まれます。

Anthropic的なハーネス観の面白さは、モデルが強くなるほど、ハーネス側の責任をどう移すかという問題も含んでいる点です。  
つまり、ハーネスは固定物ではなく、モデル能力の変化に合わせて掃除・再設計されるものです。

## 5. LangChain：ハーネス変更だけでも性能は変わる

LangChainは、測定可能な性能改善としてのハーネスチューニングを重視します。

特徴は次のようなものです。

- traces
- middleware
- self-verification hooks
- loop detection
- reasoning-budget heuristics

ここで重要なのは、「モデルを変えなくても、ハーネスを変えるだけで結果が大きく変わる」というベンチマーク志向の見方です。

この立場は、ハーネスエンジニアリングを抽象的な設計論ではなく、観測・改善・評価の対象にします。  
どこでループしたか、どの検証フックが効いたか、推論予算をどう使ったかを見ることで、エージェントの振る舞いを改善する。

## 6. Martin Fowler：共通語彙を与える

Martin Fowlerの貢献は、実装パターンそのものというより、語彙と概念分解です。

特徴的な軸は次の通りです。

- feedforward vs feedback
- computational controls vs inferential controls
- harnessability

この語彙は、他の議論を整理するために便利です。

たとえば、ドキュメントや計画、テンプレートは「行動前のガイド」、つまり feedforward 的な制御と見なせます。  
一方で、テスト、リンター、AIレビュー、実行確認は「行動後の補正」、つまり feedback 的な制御です。

この言葉があると、「うちはプロンプトを頑張っています」ではなく、  
「行動前の誘導はあるが、行動後の検証が弱い」  
「推論的な判断に頼りすぎて、計算可能なチェックが足りない」  
という議論ができます。

## 7. Gota：目的から逆算し、増やしすぎない

Gotaの焦点は、ハーネスの変更容易性と優先順位づけです。

特徴は次の通りです。

- purpose-first design
- structure before instruction
- explicit human intervention points
- pruning by business purpose

ここで大事なのは、「ハーネスを増やせば良い」という発想へのブレーキです。

何をハーネスにエンコードすべきか。  
どの制約は本当に事業目的に効いているのか。  
どこで人間が介入すべきなのか。  
構造で解ける問題を、長い指示文で解こうとしていないか。

Gotaの見方は、ハーネスエンジニアリングを「制御を足す技術」ではなく、**何を固定し、何を変えやすく残すかの判断**にします。

## 8. r.kagaya / Asterminds：ルールファイルを超え、プロセス全体へ

r.kagaya / Astermindsは、AIネイティブな開発プロセス設計を、単なる `CLAUDE.md` やスキルの話より上位に置きます。

特徴は次の通りです。

- hybrid orchestration
- eval dashboards
- trigger expansion
- self-improve loops
- provider-agnostic investment
- builder harnesses と user harnesses のスコープ分離
- `CLAUDE.md` / skills は user-harness design の一象限にすぎないという主張

この整理の強みは、ハーネスの層を分けることです。

エージェントを作る側のハーネスと、エージェントを使う側のハーネスは違う。  
また、ルールファイルだけを整えても、ワークフローの入口、測定、評価ダッシュボード、自己改善ループ、組織プロセスがなければ、ハーネス全体としては不十分かもしれない。

この見方は、「AIに読ませるファイルを作ったら終わり」という誤解を避ける助けになります。

## 9. Algomatic / AI-DLC：チームもハーネスされる

Yusuke Shibuya / Algomaticの焦点は、社会技術的なハーネス設計です。

- product principles
- design systems
- remote test runtime
- QA criteria
- models と teams の両方に対するハーネス

ここでは、ハーネスはモデルだけのものではありません。  
チームがどう判断し、どう品質を揃え、どうテストし、どうプロダクト原則に従うかもハーネスです。

yuukiyo / AI-DLCは、AIネイティブなプロセス適応を、アジャイル実践からの進化として捉えます。

- reduced role fragmentation
- flow efficiency
- human guardrails / approvals
- Scrumやfeature-team methodsからの実用的な持ち越し

この整理では、AI時代の開発プロセスは、従来のアジャイルを完全に捨てるものではありません。  
むしろ、役割分担やフロー効率、人間の承認ポイントを再設計しながら進化するものです。

## 10. Mitchell Hashimoto：日々の個人ワークフローに現れるハーネス

Mitchell Hashimotoの焦点は、実務者のワークフロー採用です。

特徴は次の通りです。

- bounded tasks
- background agents
- verification
- repeated failure turned into harness change

ここでは、ハーネスエンジニアリングは大規模なプラットフォーム設計ではなく、毎日の作業習慣として現れます。

何度も失敗する作業がある。  
それを「モデルが悪い」で終わらせず、検証手順やタスクの切り方、背景エージェントの使い方、レビュー方法を変える。  
この小さな反復もハーネスエンジニアリングです。

# 実務での使いどころ

## 1. まず「どの層のハーネス」を作っているのかを明確にする

実務では、いきなり全部を作ろうとしない方がよいです。  
KBの比較から見ると、ハーネスには少なくとも次の層があります。

- リポジトリ構造のハーネス
- ドキュメント・計画のハーネス
- テスト・リンター・構造チェックのハーネス
- レビュー・評価エージェントのハーネス
- セッション継続・引き継ぎのハーネス
- クラウド実行・権限・監査のハーネス
- チームプロセス・承認のハーネス
- 測定・トレース・改善ループのハーネス

OpenAI的に始めるなら、エージェントが迷わないリポジトリとドキュメントを作る。  
LangChain的に始めるなら、トレースや検証フックを入れて測る。  
Anthropic的に始めるなら、長時間タスクの引き継ぎやコンテキストリセットを設計する。  
Gota的に始めるなら、そもそも何をハーネス化すべきかを事業目的から選ぶ。

## 2. 「計画」を一時的な会話ではなく成果物にする

関連ノート「Coding-Agent Harness Patterns」では、`plan.md` のようなセッション単位の計画ファイルが、耐久性のある引き継ぎ契約として紹介されています。

計画を成果物にすると、次の利点があります。

- 意図が検査可能になる
- コンテキストリセットに強くなる
- レビュー担当者と認識を合わせやすい
- 検証エージェントが批評できる対象になる

これは、OpenAIの structured plans や Anthropicの structured handoffs ともつながります。

## 3. 評価者を分ける。ただし過剰なマルチエージェント化には注意する

マルチエージェントオーケストレーションの基本は、計画、生成、評価、実行を分けることです。

ただし、関連ノートでは、エージェントを増やせば常に良いわけではないとも整理されています。  
多くの推論タスクでは、一時的に複数の思考軌跡を出し、それを一つの熟慮者が批評・統合するだけでも、マルチエージェントの利点の一部を得られる可能性があります。

実務上は、いきなり複雑なエージェント組織を作るよりも、

- 実装者とは別のレビュアーを置く
- 計画だけ別モデルや別エージェントに批評させる
- 危険な変更だけ人間承認にする

といった小さな分離から始める方が扱いやすいです。

## 4. デプロイゲートを明示する

エージェントが速くなるほど、レビューやデプロイの安全性が重要になります。

実務では次のゲートを考える必要があります。

- マージ前：構造チェック、リンター、テスト、アーキテクチャ制約
- デプロイ前：E2Eテスト、スモークテスト、環境確認、ロールバック準備
- 人間介入：破壊的操作、曖昧な修正、高リスク領域
- デプロイ後：観測、回帰検出、トレース確認、継続的クリーンアップ

これは、ハーネスを「コードを書く前後」だけでなく、「本番に近づくほど強くなる安全装置」として見る発想です。

# 誤解しやすい点・論点

## 誤解1：ハーネスエンジニアリングはプロンプト改善の別名である

違います。  
プロンプトやルールファイルは一部ですが、ハーネス全体ではありません。

KB上では、`CLAUDE.md` や skills は user-harness design の一象限にすぎない、という整理もあります。  
重要なのは、構造、検証、実行環境、測定、レビュー、権限、チームプロセスまで含めて見ることです。

## 誤解2：ハーネスは多ければ多いほど良い

これも危険です。

Gotaの観点では、事業目的から外れた制御は剪定すべきです。  
ハーネスが複雑になりすぎると、変更しづらくなり、エージェントにも人間にも扱いづらくなります。

問いは「何を足すか」だけではありません。  
「何を固定しないか」「何を人間判断に残すか」も重要です。

## 誤解3：マルチエージェント化すれば自動的に賢くなる

複数エージェントは専門化を可能にしますが、調整コストも増えます。

Planner / Generator / Evaluator の分離は有効なパターンですが、すべてのタスクに恒久的な複数エージェント構成が必要なわけではありません。  
一時的な検証者、独立レビュー、計画批評だけで十分な場合もあります。

## 誤解4：クラウドエージェントはローカルCLIを遠隔実行しているだけである

Cognition / Devinの整理では、それは不十分です。

クラウドエージェントには、隔離、スナップショット、ウォームプール、権限継承、監査証跡が必要になります。  
非同期で動き続け、組織の権限境界をまたぎうる存在だからです。

## 誤解5：モデルが強くなればハーネスはいらなくなる

Anthropicの整理では、モデル能力が上がるほど、ハーネス側の責任は変化します。  
一部はモデルに移るかもしれませんが、長期作業、権限、監査、レビュー、組織プロセスの問題は残ります。

むしろ、モデルが速くなるほど、ゲートや観測、引き継ぎの重要性が増す場面があります。

# NotebookLMが会話で拾うとよい問い

## 比較のための問い

- OpenAIとAnthropicのハーネス観は、どこが似ていてどこが違うのか。
- Cursorが示している「ハーネスのプロダクト化」とは何か。
- Cognition / Devinの議論は、なぜプロンプトではなくランタイム基盤に寄っているのか。
- LangChainの測定志向は、他の設計論とどう補完関係にあるのか。
- Martin Fowlerの feedforward / feedback という語彙を使うと、各社の主張はどう整理できるか。
- Gotaの purpose-first design は、ハーネス過剰をどう防ぐのか。
- r.kagaya / Astermindsの builder harness と user harness の分離は、何を見えやすくするのか。
- AlgomaticやAI-DLCの議論は、なぜ「チームのハーネス」を強調するのか。

## 実務導入のための問い

- 自分たちの開発チームで最初に作るべきハーネスは何か。
- 失敗が多いのは、行動前の指示不足か、行動後の検証不足か。
- ドキュメントは人間向け説明に留まっているか、それともエージェントが使える地図になっているか。
- テストやリンターは、エージェントがすぐ使える信号になっているか。
- 計画や進捗は、会話の中だけで消えていないか。
- レビューや承認の速度は、エージェントの生成速度に追いついているか。
- 何度も起きる失敗を、個別注意ではなくハーネス変更に変換しているか。

## 論点を深める問い

- ハーネスはプロダクトの一部なのか、開発組織の内部インフラなのか。
- どこまでを機械的にチェックし、どこからを人間判断にするべきか。
- モデルが進化したとき、古いハーネスは技術的負債になるのか。
- エージェントにとって読みやすいリポジトリと、人間にとって読みやすいリポジトリは同じなのか。
- 長時間エージェントの記憶や進捗管理は、どこまで自動化できるのか。
- ハーネスの改善効果をどう測るべきか。

# NotebookLMに期待する話し方メモ

- 冒頭では、「ハーネスエンジニアリングはプロンプト術ではなく、エージェントの周囲の設計である」と明確に言ってほしい。
- ベンダー比較を単なるカタログにせず、「それぞれがどの失敗モードを重視しているか」という観点で話してほしい。
- OpenAIはリポジトリと機械的制約、CursorはプロダクトUIと訓練基盤、Cognition / Devinはクラウド実行基盤、Anthropicは長時間エージェントと引き継ぎ、LangChainは測定、Fowlerは語彙、Gotaは目的と剪定、というふうに対比してほしい。
- 「ハーネスを増やせば良いわけではない」という注意点を必ず入れてほしい。
- 実務者向けには、`plan.md` のような計画成果物、テスト・リンター、レビュー分離、デプロイゲートから始める話に落としてほしい。
- トーンは、流行語解説ではなく、「AIエージェント時代の開発組織をどう設計するか」という深めの議論にしてほしい。
