# 今日のランダムテーマ  
## 長く走るエージェントをどう「続ける／戻す／捨てる／任せる」か  
### Context Management Decision Map を読む

今日のテーマは、長時間動くエージェントシステムにおける **コンテキスト管理の意思決定** です。

中心になる問いはシンプルです。

> いまのセッションをこのまま続けるべきか。  
> それとも、巻き戻すべきか、要約すべきか、別エージェントに任せるべきか、完全にリセットすべきか。

LLMエージェントの実務では、モデルの性能だけでなく、周囲のハーネス、つまりリポジトリ構造、成果物、テスト、レビュー、役割分担、承認フロー、セッション継続の仕組みが成果を大きく左右します。

その中でもコンテキスト管理は、長期作業の安定性に直結する領域です。  
エージェントが長く作業するほど、会話履歴は膨らみ、ノイズも増え、過去の誤った判断や曖昧な仮定が残りやすくなります。  
そこで必要になるのが、状況に応じて「続ける」「戻す」「消す」「圧縮する」「分担する」「作り直す」「時間をおいて再開する」といった回復・継続の手を選ぶ判断地図です。

このノートでは、`Context Management Decision Map` を出発点に、長時間エージェントを安定運用するための見方を整理します。

# なぜPodcast向きか

このテーマはPodcast向きです。理由は、単なる技術用語の説明ではなく、かなり人間的な判断に近いからです。

たとえば、仕事中に「この会議、このまま続けて大丈夫か？」と思う場面があります。  
議論が脱線しているなら、いったん前の合意点に戻る。  
情報が多すぎて混乱しているなら、要約する。  
専門的な検討が必要なら、別チームに任せる。  
もう前提が壊れているなら、資料だけ残して新しい会議を立て直す。

エージェントのセッション管理も、これに似ています。

ただし、エージェントの場合は、コンテキストウィンドウ、セッション履歴、サブエージェント、スケジュール実行、明示的なハンドオフ成果物など、機械的な設計要素が絡みます。  
つまりこのテーマは、日常的な判断の比喩で話せる一方で、実際にはハーネス設計やランタイムアーキテクチャの問題でもあります。

Podcastでは、以下のような対比が会話を生みやすいです。

- 「今の流れを信じる」か「いったん断ち切る」か
- 「記憶を残す」か「記憶を消す」か
- 「人間が制御する」か「ハーネスに判断を委ねる」か
- 「一つのエージェントに粘らせる」か「役割分担する」か
- 「定期的に見に行く」か「必要なときに起こす」か

単なる機能比較ではなく、「失敗からどう回復するか」「長期作業でどう正気を保つか」という話にすると、自然で深い会話になりやすいテーマです。

# 背景と文脈

## エージェント性能はモデル単体で決まらない

関連ノート `Agent Harness Landscape` では、中心命題として次の考え方が示されています。

> エージェントの性能はモデルだけでなく、モデルの周囲にあるハーネスに大きく依存する。

ここでいうハーネスには、次のようなものが含まれます。

- 行動前のガイダンス  
  - 短い地図のような指示
  - アーキテクチャ文書
  - 明示的な計画
  - テンプレートやスキル
- 行動後の補正
  - テスト
  - リンター
  - 構造チェック
  - AIレビューや評価エージェント
- セッションをまたぐ継続性
  - 進捗ログ
  - コミット履歴
  - feature list
  - initializer agent の出力
  - context reset
- 自律性の制御
  - approval policy
  - human-in-the-loop checkpoint
  - 機械的に検査できる不変条件
- 役割分担
  - planner
  - generator
  - evaluator
  - maintenance / cleanup agent

コンテキスト管理は、このうち特に「セッションをまたぐ継続性」と「役割分担」に深く関係します。

## 長時間エージェントでは、継続そのものが難しい

短いタスクであれば、会話履歴をそのまま使っても問題が見えにくいかもしれません。  
しかし、作業が数時間、数日、あるいはそれ以上に伸びると、セッション履歴そのものがリスクになります。

- 過去の失敗した仮説が残る
- すでに不要な細部が混ざる
- 途中の議論が長くなりすぎる
- 現在の目的と過去の文脈がずれる
- エージェントが「何を信じるべきか」を見失う

Anthropic の長時間ハーネス関連の文脈では、明示的なハンドオフ成果物や、インクリメンタルな進捗が、リセットを実用可能にするとされています。  
つまり、リセットは単なる「やり直し」ではありません。  
うまく設計された成果物があれば、新しいセッションでも作業状態を再構築できる、という発想です。

## コンテキスト管理はチャットの作法ではなく、設計問題になる

`Context Management Decision Map` では、Etisha Garg / OpenClaw under the hood の貢献として、セッション解決、リセットポリシー、追記型トランスクリプト、active-run queueing、prompt assembly などが挙げられています。

ここから見えてくるのは、コンテキスト管理が「チャット画面でどう振る舞うか」だけではない、ということです。

それは Gateway や runtime architecture の問題にもなる。  
どのセッションを引き継ぐのか。  
どの履歴を使うのか。  
いつリセットするのか。  
複数の実行をどう並べるのか。  
プロンプトをどう組み立てるのか。

長時間エージェントの安定性は、こうした裏側の設計にも依存します。

# 中心アイデア

## 7つの基本ムーブ

`Context Management Decision Map` は、長く走るエージェントシステムで使える主要な回復・継続手段を、7つに整理しています。

### 1. Continue：続ける

現在のセッションとコンテキストを、そのまま維持する方法です。

向いているのは、セッションがまだ一貫していて、次のタスクが直前の作業と密接に関係している場合です。

これは最も自然で、コストも低い選択です。  
ただし、現在のコンテキストが信頼できることが前提です。

問い：

- いまの会話履歴はまだ整理されているか？
- 次のタスクは直前の流れと直接つながっているか？
- 過去の誤解やノイズが判断に影響しなさそうか？

### 2. Rewind：巻き戻す

失敗した枝を取り除き、最後にうまくいっていた地点から再開する方法です。

向いているのは、現在の推論の枝が明らかに間違っているが、セッション全体にはまだ有用な周辺情報が残っている場合です。

Continue と Reset の中間にある選択肢だと言えます。  
全部捨てるほどではないが、このまま進むと失敗が積み上がる。  
そんなときに「最後の健全な地点」に戻る。

問い：

- 間違っているのは一部の推論枝だけか？
- その前の状態には、まだ有用な情報があるか？
- 要約するより、誤った枝を切ったほうが早いか？

### 3. Clear：消す

セッションコンテキストを明示的に消し、成果物からやり直す方法です。

向いているのは、人間が「何が見えているか」を厳密に制御したい場合です。

ここで重要なのは、「消す」と「何も残さない」は同じではないことです。  
Clear は、セッション履歴を消すが、明示的な成果物を起点にする。  
つまり、人間が残したいものだけを見える状態にするための操作です。

問い：

- 今の履歴に見せたくないノイズや誤情報があるか？
- 新しい作業者に、どの成果物だけを読ませたいか？
- 暗黙の文脈ではなく、明示的な資料で始めたいか？

### 4. Compact：圧縮する

現在のセッションを短い要約に凝縮する方法です。

向いているのは、作業内容はまだ関連しているが、生の履歴が大きすぎる、またはノイズが多すぎる場合です。

Compact は、継続とリセットの間にある非常に実務的な手です。  
全部捨てるほどではない。  
しかし、そのまま読み続けるには重すぎる。  
そこで、必要な状態だけを要約して残す。

問い：

- 生ログではなく、要約で十分か？
- 重要な判断、未完了タスク、制約が要約に入っているか？
- 圧縮によって失われる微妙な文脈は問題にならないか？

### 5. Delegate：委任する

ノイズが多い、または専門的なサブタスクを、サブエージェントや別ワーカーに任せる方法です。

向いているのは、そのサブタスクが親コンテキストを汚染すべきではない場合です。

この発想は、マルチエージェント設計にもつながります。  
planner / generator / evaluator の分離、あるいは評価エージェントの分離は、主セッションの文脈を保護する意味も持ちます。

問い：

- この作業は親セッションに全部持ち込むべきか？
- 専門的な検討だけを別に走らせたほうがよいか？
- 結果だけをハンドオフ成果物として戻せば十分か？

### 6. Reset：作り直す

明示的なハンドオフ成果物をもとに、新しいセッションで状態を再構築する方法です。

向いているのは、現在のセッションに drift や hidden confusion が蓄積しすぎた場合です。

Reset は、失敗ではなく、長時間エージェントの基本技術として捉えるべきです。  
ただし、リセットが機能するには、作業状態を再構築できる成果物が必要です。

関連する実務パターンとしては、`plan.md` のようなセッションスコープの計画ファイル、進捗ログ、feature list、仕様とテストの同期などがあります。

問い：

- 今のセッションには、見えない混乱が蓄積していないか？
- ハンドオフ成果物だけで再開できるか？
- 新しいセッションに持ち込むべき最小情報は何か？

### 7. Loop / wake：ループまたは起床

スケジュールされたループや Monitor 的な wakeup を使い、時間をおいて作業を再訪する方法です。

向いているのは、定期的なメンテナンス、長い待ち時間、状態依存のフォローアップです。

これは「人間が何度も見に行く」代わりに、ハーネス側が再訪のタイミングを持つ考え方です。  
Anthropic の dynamic /loop docs は、scheduled loop を明示的なハーネスプリミティブとして示し、Monitor-based wakeup への方向性を示しています。

問い：

- このタスクは、今すぐ続けるべきか、後で再訪すべきか？
- 定期ループに向いているか？
- それとも、特定の状態変化で起こすほうがよいか？

# 実務での使いどころ

## 1. 開発ワークフローにエージェントを導入する場合

`Research OS Query Paths` では、「エージェントを開発ワークフローに導入したい」という問いに対して、`Agent Harness Landscape` から入り、ハーネス設計、承認ポリシー、エージェントに認識しやすいリポジトリなどを確認する道筋が示されています。

このとき、コンテキスト管理は最初から設計対象にすべきです。

- 作業計画はどこに残すのか
- セッションをまたぐ進捗はどう記録するのか
- いつレビューを挟むのか
- どのタイミングでリセットするのか
- サブエージェントに何を任せるのか

単に「良いプロンプトを書く」だけでは、長期の安定性は作れません。

## 2. コーディングエージェントの品質を安定させる場合

`Coding-Agent Harness Patterns` では、`plan.md` のような計画ファイルを durable handoff contract として使うことが強調されています。

これは Reset や Compact と相性がよいです。

なぜなら、セッション履歴が壊れても、計画ファイルや進捗ログ、仕様、テストが残っていれば、新しいセッションが状態を再構築しやすくなるからです。

また、Verifier split も重要です。  
全部を一つのエージェントにやらせるのではなく、計画レビューや実装レビューだけを別のエージェントに任せる。  
これにより、自己評価バイアスを減らしつつ、完全なマルチエージェント運用ほどの複雑さは避けられます。

ここでの Delegate は、単なる分業ではありません。  
親コンテキストを汚さず、評価や専門検討を別窓で行うための設計です。

## 3. レビュー負担を下げたい場合

ハーネス設計では、テスト、リンター、構造チェック、ブラウザ検証、AIレビューなどの feedback controls が重要です。

コンテキスト管理とレビューはつながっています。

- ノイズだらけのセッションでは、レビュー観点がぶれやすい
- 失敗した推論枝を残したままだと、修正方針も混乱しやすい
- Compact が雑だと、レビューに必要な判断根拠が消える
- Reset しても、ハンドオフ成果物が弱いと、同じ失敗を繰り返す

したがって、「レビューを楽にする」には、レビュー時点のコンテキストをどう整えるかも含めて設計する必要があります。

## 4. 長時間エージェントを安定運用したい場合

`Research OS Query Paths` では、「long-running agents を安定させたい」という問いに対して、`Context Management Decision Map` を出発点にする道筋が示されています。

出力形として期待されるのは、continuation / reset / delegate の decision tree です。

つまり実務では、次のような簡易判断表が役立ちます。

| 状況 | 選ぶ手 |
|---|---|
| 文脈がまだ信頼でき、次タスクも近い | Continue |
| 推論の一枝だけが明確に失敗した | Rewind |
| 履歴を人間が厳密に制御したい | Clear |
| 生履歴が重いが作業は有効 | Compact |
| 専門的・ノイズの多い副作業がある | Delegate |
| セッション全体に drift や hidden confusion がある | Reset |
| 定期的・状態依存で再訪すべき | Loop / wake |

# 誤解しやすい点・論点

## 誤解1：Reset は負けではない

Reset は「エージェントが失敗したから全部やり直す」という消極的な操作に見えがちです。  
しかし長時間作業では、リセット可能性そのものが設計品質です。

明示的なハンドオフ成果物、進捗ログ、feature list、計画ファイルがあれば、Reset は安定運用の一部になります。

論点：

- Reset を例外処理と見るか、通常運用の一部と見るか
- Reset に耐える成果物を最初から作っているか

## 誤解2：Compact すればいつでも大丈夫ではない

Compact は便利ですが、万能ではありません。

要約は圧縮である以上、何かを捨てます。  
重要な判断根拠や曖昧な未解決事項を落とすと、見かけ上は整理されても、後で問題が再発します。

`Context Management Decision Map` の Open Questions でも、Compact が失敗し Reset のほうがよい場合を予測するシグナルが問われています。

論点：

- どの情報は要約してよいのか
- どの情報は成果物として明示的に残すべきか
- Compact ではなく Reset を選ぶべき兆候は何か

## 誤解3：Delegate は高級なマルチエージェントだけの話ではない

Delegate というと、大規模なマルチエージェントシステムを想像しがちです。  
しかし `Coding-Agent Harness Patterns` では、完全なマルチエージェント運用でなくても、第二のエージェントやモデルをレビュー用途に使うだけで evaluator の利点を一部得られるとされています。

つまり、Delegate は小さく始められます。

- 計画だけレビューしてもらう
- 実装だけレビューしてもらう
- ノイズの多い調査だけ別に任せる
- 結果だけ親セッションに戻す

論点：

- どの作業を親コンテキストから隔離すべきか
- どの結果だけを戻すべきか
- evaluator を分けると、どの程度レビュー負担が下がるか

## 誤解4：Loop は「ただの定期実行」ではない

Loop / wake は、単なる cron 的な定期実行としてだけ見ると浅くなります。

重要なのは、長い待ち時間や状態依存のフォローアップを、ハーネスのプリミティブとして扱う点です。  
毎回人間が思い出して確認するのではなく、システムが再訪する。

Open Questions では、どの recurring tasks が loops に向き、どれが event-driven wakes に移るべきかが問われています。

論点：

- 定期的に見に行くべきタスクは何か
- 状態変化で起こすべきタスクは何か
- ループがノイズを増やす場合はあるか

## 誤解5：これはチャットの整理術ではなく、ハーネス設計である

Continue, Rewind, Clear, Compact だけを見ると、チャットUIの操作に見えるかもしれません。  
しかし関連ノート群では、コンテキスト管理はハーネス設計、ランタイム、成果物設計、評価分離、承認フローと結びついています。

特にベンダー比較では、Anthropic が長時間エージェント、構造化ハンドオフ、initializer agents、context resets、planner/generator/evaluator split、scheduled tasks、skills、MCP などを強調していると整理されています。

この文脈では、コンテキスト管理はプロダクトの中に組み込まれる仕組みでもあります。

# NotebookLMが会話で拾うとよい問い

## 基本の問い

- 長時間エージェントにとって、なぜ「続ける」だけでは不十分なのか？
- Continue と Reset のあいだに、Rewind や Compact がある意味は何か？
- Clear と Reset はどう違うのか？
- Compact はどんな場面で有効で、どんな場面で危険か？
- Delegate はなぜ親コンテキストの汚染を防ぐのか？

## 実務判断の問い

- いまのコンテキストが「まだ信頼できる」と判断する基準は何か？
- 失敗した推論枝だけを切る Rewind は、どんな失敗に向いているか？
- Reset を可能にするハンドオフ成果物には何が必要か？
- `plan.md` や進捗ログは、コンテキスト管理にどう効くのか？
- レビューエージェントを分けると、何が改善されるのか？

## 対立軸として面白い問い

- 要約して残すべきか、いっそ新しいセッションに移るべきか？
- 長い記憶は強みなのか、リスクなのか？
- 人間が厳密に見える情報を制御すべき場面はどこか？
- ループで定期的に見るべきか、イベント駆動で起こすべきか？
- モデルが賢くなるほど、ハーネスの責任は減るのか、それとも形を変えるのか？

## 未解決問題として拾うとよい問い

`Context Management Decision Map` の Open Questions は、そのまま会話の深掘りポイントになります。

- Compact が失敗し、Reset のほうがうまくいく兆候は何か？
- 回復手段として Rewind と Compact をどう使い分けるべきか？
- recurring tasks のうち、どれを loops にし、どれを event-driven wakes にすべきか？

# NotebookLMに期待する話し方メモ

- まず「長く働くエージェントは、記憶を持つほど賢くなる一方で、記憶に汚染される」という直感的な導入にする。
- Continue / Rewind / Clear / Compact / Delegate / Reset / Loop-wake を、機能一覧ではなく「現場での判断カード」として紹介する。
- 会話では、会議やプロジェクトの引き継ぎにたとえると自然。
- 特に強調してほしいのは、Reset は失敗ではなく、明示的なハンドオフ成果物があるから可能になる設計上の選択肢だという点。
- Compact の便利さと危険性を対比してほしい。要約は整理にもなるが、重要なニュアンスを失う可能性もある。
- Delegate は大規模マルチエージェントだけでなく、レビュー役を分ける小さな実務パターンとして話す。
- 最後は、「コンテキスト管理はチャットの掃除ではなく、長時間エージェントを運用するためのハーネス設計である」というまとめにする。
- トーンは、技術解説だが抽象論に寄りすぎず、「明日の開発ワークフローでどう判断するか」という実務感を保つ。
