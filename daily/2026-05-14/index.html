今日のランダムテーマ：Coding-Agent Review Rubric

なぜPodcast向きか

今回のテーマは「Coding-Agent Review Rubric」、つまりコーディングエージェントが出した変更を、何を軸にレビューするかという軽量な評価基準である。

一見すると、これは単なるチェックリストに見える。だが、KB全体の文脈で見ると、かなり重要な位置にある。なぜなら、エージェント時代の開発では「コードを書く能力」だけでなく、書かれたコードをどう見極めるかがボトルネックになるからだ。

エージェントはコード、設定、修正案、PR、テスト、ドキュメントを高速に生成できる。しかし、その速度が人間の理解・レビュー・運用安全性を上回ると、問題は「モデルがたまに間違える」ことではなく、レビュー体制そのものが崩れることになる。

このルーブリックは、その崩壊を防ぐための小さな部品として読める。

Podcast向きの理由は、次のような対立軸があるからだ。

  • 速く作ることと、理解可能な状態を保つこと
  • 自動チェックで判断できる品質と、人間や評価エージェントが見るべき品質
  • 汎用的なレビュー基準と、リポジトリ固有の基準
  • 短いチェックリストの再利用性と、長いチェックリストの精密さ
  • エージェントに任せる範囲と、人間が止めるべき境界

つまり、このノートは「レビューのチェックリスト」ではなく、エージェント開発における信頼性の設計図として語れる。

背景と文脈

KBでは、エージェントの性能はモデル単体では決まらない、という見方が繰り返し出てくる。重要なのは、モデルの周囲にあるものだ。

たとえば：

  • リポジトリ構造
  • テスト
  • lint
  • pre-commit
  • ドキュメント
  • 計画ファイル
  • ハンドオフ成果物
  • レビュー手順
  • 承認ポリシー
  • 独立した評価エージェント
  • デプロイ前後のゲート

これらをまとめて、KBでは「ハーネス」として捉えている。

「Coding-Agent Harness Patterns」では、コーディングエージェントのワークフローがハーネスエンジニアリングになる条件として、品質や信頼性がモデルへのプロンプトだけではなく、周辺の成果物・制御・レビュー・ループに依存することが述べられている。

「Agent Harness Landscape」でも同様に、エージェント性能はモデルだけではなく、ガイド、センサー、テスト、レビュー、役割分担、セッション継続性などの組み合わせで決まると整理されている。

この中で、今回の「Coding-Agent Review Rubric」は、特にレビュー判断の共通言語を提供するものだ。

レビューには、機械的に確認できるものと、そうでないものがある。

機械的に確認できるもの：

  • lint
  • typecheck
  • tests
  • smoke tests
  • formatting
  • 構造チェック
  • リポジトリ固有の machine-checkable invariants

一方で、機械だけでは完全には判断しづらいものもある。

  • 要件を本当に満たしているか
  • 変更範囲は過剰でないか
  • 危険な副作用はないか
  • 設計は理解可能か
  • 将来の保守者が説明できるか
  • そのリポジトリの文脈に合っているか
  • 生成されたコードが、人間のメンタルモデルを置き去りにしていないか

ルーブリックは、この後者を扱うための道具である。

中心アイデア

このノートの中心アイデアは、コーディングエージェントの出力をレビューするときに、次の6つの軸で見る、というものだ。

1. Correctness

2. Safety / Blast Radius

3. Maintainability / Simplicity

4. Test Adequacy

5. Repo Fit / Harness Fit

6. Comprehension / Reviewability

それぞれを日本語で整理すると、次のようになる。

1. Correctness：要求を満たしているか

最初の軸は、単純だが最も基本的な「正しさ」である。

見るべき問いは：

  • その変更は、意図された要求を満たしているか
  • 既存の挙動を壊していないか
  • エッジケースや失敗経路を明らかに誤って扱っていないか

ここで重要なのは、正しさは「テストが通ったか」だけではないという点だ。テストが通っていても、要求の解釈がずれていれば正しくない。逆に、要求は満たしていても、既存の挙動を壊していれば正しくない。

この軸は、KBの「Eval and Review Reliability」にある「Task-fit review」と重なる。つまり、作られたものが本当にタスクに合っているかを見る段階である。

2. Safety / Blast Radius：危険な副作用はないか

次の軸は、安全性と影響範囲である。

見るべき問いは：

  • 変更範囲はタスクに対して比例しているか
  • 危険な副作用を生まないか
  • 権限ミス、データ破壊、危険な自動化挙動につながらないか
  • ロールバックや封じ込めの懸念は妥当か

ここで出てくる「Blast Radius」は、変更が失敗したときにどこまで被害が広がるかという見方である。

「Agent Deployment Gates and Operational Risk」では、エージェントが人間のレビュー速度を超えてコードや設定を生成できるようになると、明示的なデプロイゲートが必要になるとされている。これは、単にバグを見つけるためではなく、システムレベルのレビュー崩壊を防ぐためである。

このルーブリックの安全性軸は、その前段にある。レビュー時点で「これはタスクに対して大きすぎる変更ではないか」「一見ローカルに見えて、実は危険な範囲に触っていないか」を見る。

3. Maintainability / Simplicity：保守可能で、単純か

3つ目は、保守性とシンプルさである。

見るべき問いは：

  • 変更は理解しやすく、局所的にまとまっているか
  • 責務はきれいに分かれているか
  • 不要な複雑さを避けているか

エージェントは、動くコードを素早く生成できる。しかし、速く生成されたコードが常に保守しやすいとは限らない。むしろ、生成量が増えるほど、人間が構造を把握できなくなる危険がある。

関連ノートでは、Lars Fayeの観点として、生成スループットがチームの理解能力を超えると「comprehension debt」、つまり理解負債が生まれることが示唆されている。保守性の軸は、この負債を早めに検知するための視点でもある。

ここでのポイントは、「賢そうな実装」よりも「説明できる実装」が重要になる場面が多いということだ。

4. Test Adequacy：テストは十分か

4つ目は、テストの妥当性である。

見るべき問いは：

  • テストは変更に対して比例しているか
  • ハッピーパスだけでなく、それ以外も見ているか
  • 主張されている挙動は、可能な範囲で決定的なチェックにより実際に検証されているか

「Eval and Review Reliability」では、信頼性レイヤーの最初に「Deterministic checks first」が置かれている。つまり、semantic review、意味的なレビューを行う前に、lint、typecheck、tests、smoke tests、formatting、構造チェックなどの安いチェックを先に通すべきだ、という考え方である。

このルーブリックのテスト軸は、単に「テストがあるか」ではない。テストが変更のリスクに見合っているかを見る。

たとえば：

  • 小さな表示文言の変更に大規模テストは不要かもしれない
  • 一方、挙動変更や失敗経路の修正なら、ハッピーパスだけでは足りない
  • 主張だけでなく、チェックが実際にその主張を行使しているかが重要になる

エージェントワークフローでは、テストやlintは「人間の代わり」ではなく、人間や評価エージェントの負担を減らす前処理として機能する。

5. Repo Fit / Harness Fit：リポジトリとハーネスに合っているか

5つ目は、リポジトリ適合性とハーネス適合性である。

見るべき問いは：

  • 既存の命名、構造、アーキテクチャ、慣習に合っているか
  • plan や handoff artifacts に記録された意図と整合しているか
  • lint、hook、pre-commit、review harness を尊重しているか

これは、エージェント開発で特に重要な軸である。なぜなら、エージェントは局所的なタスクに集中しすぎると、既存のリポジトリの流儀や周辺制約を見落とすことがあるからだ。

「Coding-Agent Harness Patterns」では、plan.md のようなセッション単位の計画ファイルを、耐久性のあるハンドオフ契約として使うパターンが示されている。計画は、意図を検査可能にし、コンテキストリセットやレビュー担当者の整合に役立つ。

このルーブリックは、そうした計画やハンドオフ成果物と実装が一致しているかを見る。

ここにはトレードオフもある。リポジトリ適合性を重視しすぎると、既存の慣習が過剰に保守的になり、新しい改善を阻む可能性がある。ノートでも、repo-fit 判断は、チームが定期的に慣習を見直さないと保守的になりうると指摘されている。

6. Comprehension / Reviewability：責任あるレビューが可能か

6つ目は、理解可能性とレビュー可能性である。

見るべき問いは：

  • 責任あるレビュアーが、設計と失敗モードを説明できるか
  • diff は十分小さく、深く検査できるか。分割すべきではないか
  • 生成コードが、テスト、ドキュメント、人間のメンタルモデルの維持を追い越していないか

この軸は、エージェント時代にとても重要である。

従来の開発では、コードを書く時間がレビューや理解の時間とある程度釣り合っていた。しかし、エージェントが生成速度を上げると、この均衡が崩れる。すると、見た目には開発速度が上がっていても、チームがその変更を理解できず、長期的な品質が下がる可能性がある。

「Agent Deployment Gates and Operational Risk」では、Lars Fayeの観点として、デプロイゲートは欠陥を捕まえるだけでなく、生成スループットがチームの理解・レビュー・実装スキル維持能力を超えないようにする役割も持つと整理されている。

このルーブリックの最後の軸は、まさにそこを見る。

推奨される判定形式

このノートでは、レビュー結果の形式として、次の3つが提案されている。

  • PASS
  • PASS WITH NITS
  • NEEDS FIX

これも重要である。単に長いコメントを書くのではなく、まず判定を明確にする。

さらに、出力構造としては次の形式が推奨されている。

  • Verdict
  • Critical issues
  • Nits
  • Suggested fix
  • Confidence

この構造は、人間レビューにもエージェントレビューにも使いやすい。

たとえば、エージェントにレビューさせる場合、自由形式で「感想」を書かせると、指摘の重要度が曖昧になりやすい。しかし、この形式なら、少なくとも次の区別ができる。

  • ブロッキングな問題なのか
  • 軽微な指摘なのか
  • 修正案はあるのか
  • レビュアーの確信度はどの程度か

これは「review harness」の一部として機能する。つまり、レビューを個人の勘に任せるのではなく、比較可能な出力にする。

実務での使いどころ

このルーブリックは、かなり広い場面で使える。

KBで挙げられている良いユースケースは：

  • Codex や Claude による実装後レビュー
  • PRレビュー手順
  • 人間レビュアー用のチェックリスト
  • autofix や retry loop における明示的な pass/fail 基準

実務で考えると、特に次のような場面に向いている。

1. 実装後のエージェントレビュー

エージェントがコードを変更した後、同じエージェントに「自己レビュー」させるだけでは、自己評価バイアスが残る。

「Eval and Review Reliability」では、生成と評価を分けることが重要だとされている。subagent review、adversarial review、blocking review gates、人間へのエスカレーションなどが挙げられている。

このルーブリックは、独立した評価エージェントに渡す判断基準として使える。

2. PRレビューの共通言語

人間のPRレビューでも、レビュー観点が人によってずれることがある。

ある人は設計を見る。ある人はテストを見る。ある人は命名を見る。ある人は安全性を見る。どれも重要だが、基準が共有されていないと、レビュー結果がばらつく。

このルーブリックは、レビュー時に最低限見るべき軸を揃える。

3. retry loop や autofix の停止条件

エージェントに修正を繰り返させる場合、「いつ完了とみなすか」が問題になる。

テストが通ったら終わり、という基準だけでは不足することがある。なぜなら、テストは通っても変更範囲が過剰だったり、リポジトリの設計に合わなかったり、人間が理解しづらいdiffになっていたりするからだ。

そこで、このルーブリックを pass/fail の基準として使うと、retry loop の判断が明確になる。

4. ハーネス改善の起点

「Eval and Review Reliability」では、繰り返し発生するレビュー失敗を、より強いハーネスルールに変換することが推奨されている。

たとえば：

  • よく指摘されるなら、新しいテストを書く
  • よく迷うなら、指示を明確にする
  • よく規約違反が起きるなら、repo rule や lint を追加する
  • レビュー観点が足りないなら、rubric を更新する

つまり、このルーブリックは固定されたものではなく、レビューの失敗からハーネスを育てる入口になる。

誤解しやすい点・論点

誤解1：これはテストの代わりである

違う。

このルーブリックは、テストやlintを置き換えるものではない。むしろ、決定的チェックを先に走らせたうえで、それでは判断しきれない意味的な品質を見るものだ。

KBでも、決定的チェックを先に置き、その後にタスク適合性レビューや独立評価ループを置く構造になっている。

誤解2：短いルーブリックなら常に十分である

これも違う。

ノート自身がトレードオフを認めている。短いルーブリックは再利用しやすいが、ドメイン固有の品質問題を見落とすかもしれない。長いルーブリックは具体性が上がるが、レビューコストや儀式感が増える。

ここには実務上の問いがある。

  • どこまで汎用の基準で見るべきか
  • どこからリポジトリ固有・ドメイン固有にするべきか
  • 長くなりすぎたルーブリックは、実際に使われ続けるのか

誤解3：Repo Fit は現状維持の言い訳である

Repo Fit は重要だが、危険もある。

既存の命名や構造に合うことは、エージェントがリポジトリを壊さないために必要である。一方で、既存の慣習が古くなっている場合、それに合わせ続けることが改善を妨げる可能性もある。

そのため、Repo Fit は「絶対に変えるな」ではなく、変更するなら意図と影響を説明できるかという問いとして扱うのがよい。

誤解4：レビュー可能性は主観的すぎる

たしかに、Comprehension / Reviewability は数値化しづらい。しかし、だから不要なのではない。

エージェントが生成するコード量が増えるほど、レビュー可能性は実務上の制約になる。diff が大きすぎる、責務が混ざっている、失敗モードが説明できない、テストやドキュメントが追いついていない。これらは、たとえテストが通っていても長期的には危険である。

論点：数値スコアにするべきか、カテゴリ判定でよいか

Open Questions では、各カテゴリを数値で採点するべきか、それとも categorical verdict で十分かが問われている。

数値スコアの利点：

  • 比較しやすい
  • ダッシュボード化しやすい
  • 繰り返し評価に向いている

カテゴリ判定の利点：

  • 軽い
  • 実務で使いやすい
  • 過剰な精密さを避けられる

このノートでは、まず PASS / PASS WITH NITS / NEEDS FIX という軽量な形式が示されている。これは、ハーネス部品としての再利用性を重視した設計に見える。

NotebookLMに期待する話し方メモ

  • 単なる「コードレビューのチェックリスト紹介」ではなく、エージェント時代のレビュー崩壊を防ぐハーネス部品として語ってほしい。
  • 会話では、まず「モデルが賢くなるほど、なぜレビュー基準が必要になるのか」という逆説から入ると自然。
  • 6つの軸は一気に読み上げるだけでなく、それぞれを「何を防ぐための軸か」で説明してほしい。
  • 特に強調してほしい対立軸は以下：

- テストでわかること vs レビューでしかわからないこと

- 生成速度 vs 理解可能性

- 汎用ルーブリック vs リポジトリ固有ルール

- PASS判定の軽さ vs 数値スコアの精密さ

  • 「Comprehension / Reviewability」は深掘りしてほしい。エージェントが作ったコードが動いても、人間が説明できなければ危険、という論点がPodcast向き。
  • 「Repo Fit」は保守性だけでなく、ハーネスへの適合として語るとよい。plan、handoff、lint、pre-commit、review harness とつなげる。
  • 最後は、ルーブリックを固定的な儀式ではなく、レビュー失敗からテスト・lint・repo rule を育てるためのフィードバック装置としてまとめてほしい。
