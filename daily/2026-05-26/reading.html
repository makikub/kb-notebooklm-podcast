Agent Harness Control Taxonomy

エージェントを制御する仕組みを、全部まとめて「ガードレール」と呼ぶと設計がぼやける。今日のテーマは、制御をいつ効かせるか、どう効かせるか、どの範囲に効かせるかで分解するための分類表。

Source page Markdown source GitHub
1. いつ効くか行動前、作業中、行動後で制御の役割は違う。計画、hook、testを同じ言葉で扱わない。
2. どう効くか決定的チェックと意味的判断を分ける。lintとレビューエージェントは同じ種類の制御ではない。
3. どこに効くか評価、実行、文脈、安全の4層を分けると、足りないハーネスが見えやすい。

「ガードレール」だけでは足りない

AIエージェント運用では、よく「ガードレールを入れる」と言います。便利な言葉ですが、実装を考える段階では少し粗すぎます。テストも、権限確認も、計画ファイルも、メモリも、サンドボックスも、全部ガードレールと言えてしまうからです。

今日の Agent Harness Control Taxonomy は、この曖昧さをほどくための分類です。エージェントの性能や安全性は、モデルだけでなく、その周囲にあるハーネスでかなり変わります。だからこそ、制御の種類を分けて考えないと、どこを強くすべきか分かりにくくなります。

1つ目の軸は「いつ効くか」

まず見るべきなのは、制御がいつ働くかです。行動前に進路を整える制御、作業中に介入する制御、作業後に確認する制御では、設計も得意な失敗の種類も違います。

行動前の制御には、タスク分解、計画、アーキテクチャ文書、承認ポリシー、使ってよいツールの境界、スキル、初期化用の文脈があります。これはエージェントが走り出す前に、迷いにくい地形を作る仕事です。

作業中の制御には、tool callのhook、権限チェック、途中ログ、進捗確認、リトライ制御、context追加があります。エージェントがすでに動いている最中に、危険な操作を止めたり、判断を補助したりします。

行動後の制御には、test、lint、typecheck、review、diff確認、公開URLの検証、postmortemがあります。成果物を見て、通してよいか、直すべきか、次回のルールに移すべきかを判断する層です。

2つ目の軸は「どう効くか」

次に、制御が決定的に働くのか、意味的に判断するのかを分けます。ここを混ぜると、レビュー設計が不安定になります。

決定的な制御は、同じ入力なら同じ結果を返すものです。lint、型チェック、テスト、JSON schema、ファイル存在確認、HTTP status確認、禁止パスへの書き込み拒否。これは速く、安く、何度でも回せます。

意味的な制御は、文脈を読んで判断するものです。依頼に合っているか、文章が人間向けか、公開してよい情報だけか、設計が将来の作業に耐えるか、別エージェントによるレビューが妥当か。こちらは強力ですが、曖昧さとコストがあります。

実務では、先に決定的チェックを厚くして、その後に意味的レビューを置くのが強いです。人間や評価エージェントの注意を、機械で落とせる問題に使わないためです。

4つのスコープで見る

この分類は、評価、実行、文脈、安全の4つのスコープにも分けられます。同じhookでも、何を守るかによって意味が変わります。

  • 評価ハーネス: modelや成果物を採点する仕組み。SWE-bench風のgrader、unit test、rubric、review loop。
  • 実行ハーネス: エージェントがreasoning、tool use、observation、retryを回すループ。途中介入や状態管理もここ。
  • 文脈ハーネス: instructions、AGENTS.md、memory、skills、retrieval、MCP、repo docs。モデルに何を見せ、どう振る舞わせるか。
  • 安全ハーネス: sandbox、approval、credential boundary、audit log、外部送信前の確認。被害の範囲を制御する層。

大事なのは、この4つが排他的ではないことです。たとえばスキルは文脈ハーネスですが、外部送信前の確認ルールを含むなら安全ハーネスにも関わります。テストは評価ハーネスですが、デプロイ前ゲートになれば安全側にも効きます。

memoryとskillは「文脈ハーネス」の中心にある

今回の分類で特に面白いのは、memoryとskillの位置づけです。memoryはただの過去ログではありません。次の作業で何を思い出し、何を避け、どの手順を選ぶかを変える文脈面の制御です。

skillは手続き的な記憶です。NotebookLMの音声生成、KBページ公開、X API取得、Linear記録のように、失敗しやすい手順や確認項目を、次回のエージェントが再利用できる形にします。

この2つは別物というより、同じハーネスの違う面です。memoryが「何を覚えるか」を持ち、skillが「どう動くか」を持つ。うまくつながると、会話で一度指摘されたことが、次回以降の初動に反映されます。

個人運用での使い方

MasakiのClawd運用に落とすなら、この分類はチェックリストとして使えます。何かを自動化したり、公開したり、Discordに送ったりするときに、どの層が足りないかを見る。

たとえば今回の読み物ページなら、行動前の制御は「読み物を人間向け成果物として扱う」というskillルールです。作業中の制御は、出力ディレクトリと公開スクリプトの導線確認。行動後の制御は、HTTP 200、リンク確認、git diff check、Linear/memoryへの記録です。

これを毎回その場の注意力だけでやると、抜けます。だから、スクリプト、skill、memory、Linearに分散して残す。エージェントの信頼性は、賢い返答よりも、こういう地味な足場で上がります。

分類表としての読みどころ

このノートの価値は、「エージェント制御」を大きな精神論にしないところです。制御は、導入するタイミング、判定方法、作用するスコープに分けられます。すると、今の仕組みに何が足りないかを会話ではなく設計で議論できます。

行動前の文脈が弱いなら、repo docsやskillを足す。作業中の危険操作が不安なら、approvalやhookを足す。作業後の品質が揺れるなら、testと独立レビューを足す。公開や送信が怖いなら、安全ハーネスの境界を明確にする。

「guardrailを入れる」ではなく、「どのタイミングで、どの方式で、どのスコープに効く制御を入れるか」と言えるようにする。それだけで、エージェント運用の設計精度がかなり上がる。

次に考えるとよさそうな問い

  • いまの自動化で、行動前・作業中・行動後のどこが一番薄いか？
  • 人間レビューで見ている項目のうち、決定的チェックに移せるものは何か？
  • memoryとskillのどちらに残すべき学習なのか？
  • 外部送信、公開、デプロイ前に必ず止めるべきゲートは何か？
  • 「安全」「評価」「文脈」「実行」を混同している言葉はないか？
