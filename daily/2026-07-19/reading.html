Context Management Decision Map

長く動くエージェントで一番むずかしいのは、賢く続けることだけではない。続ける、巻き戻す、消す、圧縮する、任せる、リセットする、あとで起こす。その判断を、気分ではなくハーネスの部品として扱うための読み物です。

3. 引き継ぎ artifact が鍵になる計画、進捗、検証結果、公開物が残っていれば、リセットしても仕事は失われない。

今日のテーマは「どの文脈で続けるか」

エージェントが長く働けるようになるほど、会話を続けるかどうか自体が設計問題になります。人間とのやり取り、読んだファイル、試したコマンド、途中の仮説、失敗した枝。これらは次の判断を助けますが、同時に誤解や古い前提も運びます。

Context Management Decision Map は、その場の勘で「まだ続ける」「一回やり直す」を決めないための地図です。文脈をそのまま使うのか、圧縮するのか、消すのか、別作業へ逃がすのか。長時間エージェントの信頼性は、この選択をどれだけ明示できるかで大きく変わります。

ここでの主役はモデルの記憶力ではありません。ハーネス側に、文脈を残す方法と捨てる方法の両方を持たせることです。

Continue は、まだ地図が汚れていないときに使う

一番自然なのは、そのまま続けることです。直前の作業がまだ明確で、次にやることも近く、会話の中に有用な判断材料が多いなら、continue が一番安い。読み直しも引き継ぎも少なく、作業の勢いを保てます。

ただし、continue は便利なぶん、惰性で選ばれがちです。途中で間違った前提が入り、古い方針が残り、話題が広がりすぎた状態でも続けてしまうと、エージェントは「覚えていること」に引っ張られます。

だから continue の判断基準は、「まだ作業文脈が助けになっているか」です。現在の次の一手が明快で、ユーザーの最新指示と矛盾せず、検証結果も読めている。そこまで揃っているなら、文脈は資産として使えます。

Rewind と Clear は、間違った枝を扱うための操作

失敗した枝に入ったとき、全部を捨てる必要はありません。推論の途中だけが間違っていて、周囲の状態はまだ使えるなら rewind が効きます。最後に正しかった地点まで戻り、そこから別の分岐を試す。

一方で、ユーザーが「この前提は消したい」「ここからは見える情報を絞りたい」と考えているなら clear の出番です。明示的に会話文脈を拭き取り、必要な artifact だけを入口にして始める。これは単なる忘却ではなく、見えるものを制御するための操作です。

この二つは似ていますが、目的が違います。rewind は作業の枝刈り、clear は可視状態の制御です。どちらも、長い会話の中で「何を持ち越すか」を人間が選べるようにするためにあります。

Compact は、捨てずに軽くする技術

文脈が大きくなりすぎたけれど、まだ価値がある。そんなときは compact です。会話全体を残すのではなく、決定、未完了、検証結果、変更ファイル、次の一手だけに縮める。

圧縮の難しさは、短くすることではありません。何を落としてよいかを見分けることです。冗長なログ、古い仮説、すでに解決した寄り道は落とす。反対に、ユーザーの最新要求、外部に出す成果物、未検証のリスク、失敗したコマンドは残す。

良い compact は、次のエージェントに「続きをやるための最低限」を渡します。悪い compact は、気持ちよく短くなっているだけで、復帰に必要な根拠が抜けています。ここでも、圧縮は要約テクニックではなくハーネス設計です。

Delegate は、親の文脈を汚さないために使う

専門的でノイズの多い調査、長いログの探索、別視点のレビュー、独立した検証。こうした作業を親セッションで全部抱えると、会話が膨らみ、判断軸がぼやけます。そこで delegate が効きます。

サブエージェントや別ワーカーに、狭い目的だけを渡す。親は依頼内容、戻り値、採用した判断だけを受け取る。これにより、探索のノイズは隔離され、親の文脈は意思決定に必要な情報へ寄ります。

これはマルチエージェントを派手に使う話ではありません。むしろ、親の作業面を汚さないための小さな分離です。レビュー役を分ける、ログ解析を分ける、候補出しを分ける。そのくらいでも十分に効果があります。

Reset は、成果物から立ち上がれるときに強い

セッションが長くなりすぎ、方針の drift や隠れた混乱が増えてきたら、reset を選びます。新しいセッションで、明示的な handoff artifact から状態を再構築する。

ここで重要なのは、reset ができるように普段から作業することです。PROGRESS.md、計画ファイル、コミット、検証ログ、公開URL、未完了リストが残っていれば、会話を捨てても仕事は続きます。逆に、重要な判断がチャットの奥にしかないなら、reset は危険になります。

長時間ハーネスが進捗ファイルや小さなコミットを重視する理由はここにあります。文脈を永遠に保持するより、外部 artifact から復帰できるほうが、運用としては強い。

Loop と Wake は、時間をまたぐ仕事のためにある

すべての仕事が一回のセッションで終わるわけではありません。PRのレビュー待ち、公開反映待ち、定期レポート、外部状態の監視、毎週の読み物生成。こうした作業では、loop や wake が文脈管理の一部になります。

scheduled loop は、一定の条件や時刻で作業を再開します。Monitor 型の wakeup は、新しい入力や状態変化を見て戻ってきます。どちらも、会話をずっと開いたままにするのではなく、必要なときに短い文脈で復帰する発想です。

ただし、時間をまたぐほど artifact の質が問われます。前回何を選んだか、何を公開したか、何を待っているか、次に何をすればよいか。これが残っていない loop は、ただ定期的に迷う仕組みになります。

判断地図は、ハーネス全体につながっている

このテーマは、単独の会話術ではありません。KBの Agent Harness Landscape や Coding-Agent Harness Patterns とつながっています。エージェントの性能は、モデルだけでなく、リポジトリ構造、検証、レビュー、権限、メモリ、公開後の循環で決まる。

文脈管理は、その中でも「次の判断に何を見せるか」を決める層です。見せすぎるとノイズになる。見せなさすぎると再発見になる。圧縮しすぎると根拠が消える。残しすぎると古い前提に引っ張られる。

だから実務では、判断地図をチェックリストとして使うとよいです。まだ continue でよいか。失敗枝だけ rewind できるか。clear が必要か。compact すべきか。delegate すべきか。reset できるだけの artifact はあるか。待ちがあるなら loop に逃がせるか。

長く働くエージェントに必要なのは、文脈をたくさん持つことだけではない。いつ続け、いつ捨て、いつ圧縮し、いつ別の作業面に逃がすかを選べることが、信頼できる運用を作る。

次に考えるとよさそうな問い

  • いまの作業は continue で進めるべきか、それとも compact してから続けるべきか？
  • 失敗した枝だけを rewind できるほど、直前の正しい地点は明確か？
  • 新しいセッションで reset しても復帰できる handoff artifact は残っているか？
  • 親セッションから delegate したほうがよいノイズの多い作業は何か？
  • 時間待ちや定期確認を、会話の記憶ではなく loop / wake に移せるか？
